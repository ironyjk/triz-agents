## Argument Parsing (do this FIRST)

Extract from $ARGUMENTS:

- **Contradiction**: A technical or physical contradiction to resolve, or a problem statement from which to derive applicable principles. Look for "improve X without worsening Y" or "X must be A and not-A" patterns. The entire argument text IS the contradiction if not explicitly labeled.
- **Format**: `mermaid` (default) or `ascii`. Set by `--format`.
- **Depth**: `shallow` (default, top 3-5 most relevant principles with one concept per principle) or `deep` (top 8-10 principles with multiple sub-principles and analogies from different domains). Set by `--depth`.

If Contradiction is missing or too vague, use AskUserQuestion:

```
header: "Contradiction"
question: "State the contradiction you want to resolve. Either: (1) a technical contradiction — 'Improving X worsens Y', or (2) a physical contradiction — 'Parameter must be both A and not-A'. You can also describe a problem and I will identify applicable principles."
```

## Execution

1. Read the protocol file:
   ```
   .claude/skills/triz/references/40p-protocol.md
   ```

2. Read the output format specification:
   ```
   .claude/skills/triz/references/output-format.md
   ```

3. Execute the 40 Inventive Principles protocol:
   - Phase 1: Parse and classify the contradiction (TC or PC)
   - Phase 2: If TC — use the Contradiction Matrix to identify recommended principles
   - Phase 3: If PC — apply separation principles (time, space, condition, scale) to select relevant principles
   - Phase 4: For each selected principle, generate concrete solution concepts specific to the user's domain
   - Phase 5: Rank solutions by feasibility and inventive level
   - Phase 6: Render output in requested format

4. Present the result with:
   - Contradiction restated in standard form
   - Each applicable principle: number, name, sub-principles used
   - Concrete solution concept for each principle (domain-specific, not generic)
   - Feasibility rating (High/Medium/Low) and inventive level (1-5)
   - Top recommendation with rationale
   - Suggested next step: `/triz:sufield` if interaction-level redesign needed, `/triz:ifr` to validate against ideal
